# Cordova Android 定制项目目录结构及作用

本文档用于记录和说明 `cordova-android` 源码仓库的核心目录结构及其各自的作用，帮助后续开发者快速理解平台的运作机制。

## 根目录结构概览

```text
cordova-android/
├── bin/                 # Cordova 平台创建和管理的 CLI 工具脚本
├── cordova-js-src/      # 注入到 WebView 中的平台专用 JavaScript 桥接源码
├── framework/           # Android Native 层的核心框架，被编译打包发布为基础依赖库 (AAR)
├── lib/                 # Node.js 脚本库，用于支撑 Cordova CLI (例如 prepare、build、run)
├── templates/           # 创建新 Cordova 项目时使用的 Android 工程骨架模板 (Hello World 应用模型)
├── test/                # 源码级单元测试及集成测试目录 (由于独立化已逐渐弃用)
├── docs/                # 项目相关的内部总结文档 (自建)
├── package.json         # Node 包描述文件，配置 CLI 调用的入口和 npm 依赖
└── RELEASENOTES.md      # 官方版本的更新与发布记录
```

---

## 核心目录详细说明

### 1. `framework/` (Android 核心模块)
此目录是构建 `cordova-webcontainer` 及所有底层定制的**基石**，整个项目的核心 Native 代码都在这里。  
在原生的工作流中，这部分被作为 `libs` 依赖提供给 `cordova-lib` 等容器使用。在我们基于 **“统一发布 AAR”** 的最新改造后：
- `framework/src/org/apache/cordova/` 存放所有 `PluginManager`、`CordovaWebView` 等底层类的源代码。
- `framework/src/org/apache/cordova/customer` 存放由 YUMC 及个人衍生的**拦截和调用定制化增强代码**。
- `framework/build.gradle.kts` 已被我们全方位改为脱离旧环境、支持直接使用 `maven-publish` 打包为标准 Maven 构件（AAR）至远端或本地。

### 2. `cordova-js-src/` (JS API 层)
里面包含的是由 Native 层加载的初始化逻辑。比如 `cordova.js`、`platform.js`、`exec.js` 以及插件管理器核心等机制。
在此前生成 `cordova.js` 产物时，正是 Node.js 脚本将这个目录的内容与 Cordova 核心 JS 缝合，形成注入给 H5 使用的完整 `cordova.js`。

### 3. `bin/` & `lib/` (CLI 工具链)
这是供全局 `$ cordova platform add android` / `$ cordova build android` 调用的引擎：
- `bin/create` 是用以通过模板克隆出一个独立 Android 工程实例的启动脚本。
- `lib/` 目录中的逻辑（`Api.js`、`builders/`、`prepare.js` 等）负责对生成出来的 Android 工程执行 Gradle 语法注入、AndroidManifest 合并以及对 `framework` 依赖的桥接引用等工作。

### 4. `templates/` (脚手架工程模板)
当使用 Cordova CLI 构建 `android` 平台时，系统会将 `templates/project/` 作为一个干净的 Android Gradle 完整壳工程进行复制，后续再由 `lib/` 脚本将你的 H5 代码及其配置写进该骨架里合并。里面预写好了 `app` 工程目录及入口 `MainActivity.java`。

---

> [!NOTE]
> 在我们**前端容器化 (WebContainer)** 的特殊使用场景下，实际上重点关注并迭代的仅有 `framework/`（作为提供支持库的上游）以及基于此独立发布的构建系统，传统通过 `cordova-cli` 建导打包体系的 Node 侧（`lib` 和 `bin`）往往作为生态依赖原样保留即可。
